# NOVOVM EVM 插件边界铁律（2026-03-13）

## 1. 适用范围

- 适用于 `SUPERVM` 主仓内所有 EVM 相关实现、接线与迁移文档。
- 适用于 `gateway -> node -> AOEM` 全链路。

## 2. 目录与命名铁律

1. EVM 网关实现必须放在专属目录：`crates/gateways/evm-gateway`。
2. EVM 核心与插件实现目录固定为：
   `crates/plugins/evm/core`、`crates/plugins/evm/plugin`。
3. EVM 相关实现应持续收敛到上述专属边界，不得继续散落到无链类型语义的通用目录。
4. 历史 `crates/novovm-edge-gateway` 已下线，后续不得恢复该混合命名与混合职责结构。

## 3. 通信与性能铁律

1. 对外接口可以 `HTTP/JSON-RPC`；对内必须保持二进制流水线：
   `ops_wire_v1/.opsw1 -> novovm-node -> AOEM`。
2. superVM 内部层间（D1/D2/D3/AOEM）禁止引入插件专属 RPC/HTTP 文本通信。
3. 默认极致静默，日志最小化；仅允许错误与必要汇总，调试日志必须显式开关。

## 4. 生产主线铁律

1. 不得以 gate/signal/snapshot/rc 脚本通过替代生产功能完成。
2. 迁移以规则直迁为主，不做非必要“工程化包装层”。
3. 任何新增中间层若不能证明必要性与性能收益，禁止进入主线。

## 5. 公共入口保护铁律

1. `novovm-node` 属于公共入口，非 EVM 专属代码不得擅自侵入。
2. 修改 `novovm-node` 公共方法或公共语义，必须先取得负责人手工同意。
3. EVM 专属逻辑应在 EVM 专属网关/插件侧完成，不得反向污染公共入口。

## 6. 插件隔离铁律（面向未来第三方化）

1. 主仓保留插件接口与能力边界；插件实现必须保持可剥离、可独立开源。
2. 不得形成“必须与主仓内部私有细节耦合”才能运行的插件结构。
3. 文档与代码命名应体现“可第三方独立交付”的边界，而非“主仓内置功能”。
